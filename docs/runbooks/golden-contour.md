# Golden Contour Runbook

This document freezes the currently working VPN contour exactly as it behaves in production today.

It is intentionally operational, not aspirational. It records what is proven now so that future debugging does not have to reconstruct the contour from chat history.

## Scope

This freeze-state kit covers:

- the current client -> YC entry -> DO egress -> Internet data path
- the current DO app host -> YC SSH control plane
- the canonical smoke tests for the contour
- the current invariants for generated client configs

This document does not redesign:

- Terraform
- failover / standby
- mobile app flows
- admin panel / subscription product work
- provisioning architecture

## Proven Architecture

The current working contour is:

- client -> YC entry -> DO egress -> Internet
- DO app host -> SSH -> YC for control-plane provisioning

Roles:

- YC entry
  - public WireGuard listener for clients
  - `wg-clients` is the client-facing interface
  - `wg-upstream` forwards traffic toward DO egress
- DO egress
  - receives upstream traffic from YC
  - performs NAT to the Internet
- DO app host
  - runs `app-api-1` and `app-bot-1`
  - provisions peers and generates configs

## Historical Debugging Trap

Do not treat this host-shell command on the DO app host as authoritative:

```bash
ssh -F /opt/vpn/secrets/ssh_config.app yc-vpnmgr 'echo SSH_OK user=$(whoami)'
```

This command has produced false negatives such as banner exchange timeouts while:

- bot provisioning still worked
- peers were still created
- users still connected successfully

The host shell command is therefore debug-only. It is not the canonical control-plane gate.

## Real Runtime Control-Plane Path

The actual provisioning path in code is:

1. A client-facing call triggers device creation.
2. The API handles `POST /devices/` in [internal/transport/http/router.go](/Users/antongavrilov/Desktop/workspace/vpn-backend/internal/transport/http/router.go:258).
3. That handler resolves the caller identity from `X-Telegram-ID` or `X-User-ID` in [internal/transport/http/router.go](/Users/antongavrilov/Desktop/workspace/vpn-backend/internal/transport/http/router.go:373).
4. The HTTP layer calls `application.CreateDevice.Execute(...)` in [cmd/api/main.go](/Users/antongavrilov/Desktop/workspace/vpn-backend/cmd/api/main.go:107).
5. `CreateDeviceUseCase.Execute` generates a client keypair, allocates the next client IP, and asks the VPN transport to create a peer in [internal/app/create_device.go](/Users/antongavrilov/Desktop/workspace/vpn-backend/internal/app/create_device.go:56).
6. The proxy transport executes the remote peer-management script over SSH in [internal/infra/proxy/transport.go](/Users/antongavrilov/Desktop/workspace/vpn-backend/internal/infra/proxy/transport.go:122).
7. The client config string is generated by the WireGuard config builder in [internal/infra/wireguard/client_config_builder.go](/Users/antongavrilov/Desktop/workspace/vpn-backend/internal/infra/wireguard/client_config_builder.go:52).
8. The API returns the config as `client_config`, and the bot simply relays that text and QR without rewriting it in [internal/bot/bot.go](/Users/antongavrilov/Desktop/workspace/vpn-backend/internal/bot/bot.go:415) and [internal/bot/bot.go](/Users/antongavrilov/Desktop/workspace/vpn-backend/internal/bot/bot.go:1020).

Important consequence:

- the canonical operational smoke test is a real `POST /devices/` call
- the bot `/newdevice` flow is equivalent from a provisioning perspective, but adds Telegram delivery as an extra layer
- for deterministic ops verification, the API provisioning call is the canonical smoke

## Canonical Smoke Tests

### A. Canonical control-plane smoke

Use a real disposable API provisioning request against the running app on the DO app host.

This is the source-of-truth control-plane smoke because it exercises:

- HTTP routing
- Telegram/user identity resolution
- `CreateDeviceUseCase`
- peer creation on YC
- client config generation

Example command from an operator machine, executed through the DO app host shell:

```bash
ssh "$DO_APP_USER@$DO_APP_HOST" '
  curl -sS -i \
    -H "Content-Type: application/json" \
    -H "X-Telegram-ID: <telegram_id>" \
    -d "{\"name\":\"freeze-smoke-$(date +%Y%m%d%H%M%S)\"}" \
    http://127.0.0.1:8080/devices/
'
```

Equivalent header-based form when you want to use a direct backend user id:

```bash
ssh "$DO_APP_USER@$DO_APP_HOST" '
  curl -sS -i \
    -H "Content-Type: application/json" \
    -H "X-User-ID: <user_id>" \
    -d "{\"name\":\"freeze-smoke-$(date +%Y%m%d%H%M%S)\"}" \
    http://127.0.0.1:8080/devices/
'
```

Success criteria:

- HTTP `201 Created`
- response JSON includes `client_config`
- response config contains the frozen `[Peer]` invariant above

### B. Runtime network/control-plane smoke

This is the best shell-based runtime approximation that is already proven in production debugging:

```bash
docker run --rm \
  --network "container:app-api-1" \
  -v /opt/vpn/secrets:/opt/vpn/secrets:ro \
  alpine:3.20 sh -lc '
    apk add --no-cache openssh-client >/dev/null &&
    ssh -F /opt/vpn/secrets/ssh_config.app yc-vpnmgr "echo SSH_OK user=\$(whoami)"
  '
```

Why it is valid:

- it uses the `app-api-1` network namespace
- it uses the same mounted host-side secrets
- it is the runtime-adjacent SSH path that was already proven while the contour itself worked

Why it is still secondary:

- it is still a shell approximation
- the authoritative gate is still the real `POST /devices/` path above

### C. Transport / user-plane smoke

On YC:

```bash
sudo wg show
ip rule
ip route show table 51820
ip route get 1.1.1.1 from 10.69.0.10 iif wg-clients
```

Expected:

- `wg-clients` exists
- `wg-upstream` exists
- `ip route get ... from 10.69.0.10 iif wg-clients` shows the client path leaving via `wg-upstream`

On DO egress:

```bash
sudo wg show
sudo iptables -S FORWARD
sudo iptables -t nat -S POSTROUTING
sudo sysctl -n net.ipv4.ip_forward
```

Expected:

- `wg-upstream` is up
- forwarding rules exist for client subnet traffic and return traffic
- `POSTROUTING` contains NAT / MASQUERADE for the client subnet
- `net.ipv4.ip_forward = 1`

Packet-path examples:

On YC:

```bash
sudo timeout 20 tcpdump -ni any -vv 'icmp and (host 1.1.1.1 or host 10.69.0.10)'
```

On DO egress:

```bash
sudo timeout 20 tcpdump -ni any -vv 'icmp and host 1.1.1.1'
```

On the client:

```bash
ping 1.1.1.1
```

Interpretation:

- traffic on YC but not on DO egress means trouble between YC and DO
- traffic on DO egress tunnel but not to the Internet means FORWARD / NAT trouble on DO
- no traffic on YC at all means wrong client config or no real tunnel traffic
